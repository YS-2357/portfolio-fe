# **3.** 🧾 **프로젝트 개요**

- **프로젝트 동기 및 배경**
    - 검색 트렌드가 SEO(키워드 중심)에서 GEO(질문 응답 기반)로 빠르게 전환 중
    - 이미지 중심의 기존 상세페이지는 AI 검색에 노출되지 않는 구조적 한계 존재
    - 디지털 전환에 어려움을 겪는 소상공인을 위한 상세페이지 자동 생성 도구의 필요성 제기
- **목표**
    - 상품 정보(이름, 특징, 이미지 등)를 입력하면 LLM 기반 상세페이지 + AI 이미지 생성
    - SEO·GEO 최적화된 상세페이지를 자동 생성하는 웹 서비스 개발
- **기대 효과**
    - 광고비 없이도 AI 검색에 잘 노출되는 상세페이지 생성
    - 상세페이지 제작 시간·비용 절감 → 마케팅 생산성 향상
    - 디지털 취약 계층 소상공인도 쉽게 활용 가능한 도구 제공
- **의류 특화 도메인**
    - AI가 착용샷을 만들어주는 기능에 대한 수요가 클 것으로 예상
    - 의류 전용 허깅페이스 모델이 많아 기술적으로 도전해볼 가치

# **6.** 🚧 Backend

## 6.1. Input Handler

> 프론트에서 전달된 상품 정보 및 이미지를 검증 및 전처리
> 

**Input Data**

```yaml
input:
  name: "롤프 남성 정장자켓 수트마이 양복상하의"
  category: "양복"
  price: 58000
  brand: "롤프"
  features: "꼼꼼한 박음질, 고급스러운 원단, 클래식한 디자인"
  image_path_list: ["backend/data/output/suit.png"]
```

**사용자 입력 파싱 및 유효성 검증**

- 상품명, 가격, 브랜드, 카테고리, 특징, 이미지 경로 정제
    - 공백 제거, 특수 문자 제거
- 전체 폼 데이터 파싱 및 Pydantic 스키마로 검증
    - 카테고리, 브랜드 텍스트 길이 제한 50자
    - 가격이 숫자 타입인지, 범위가 0 < price < 10000000 인지 체크
    - 상품 특징 텍스트 길이 제한 500자
    - 이미지 경로 유효한지 체크

**이미지 유효성 검사 및 리사이징 처리**

- 업로드된 단일 또는 다중 이미지 전처리
- 파일 존재, 용량, 포맷, 크기 검증
    - 단일 파일 5MB까지 허용
    - 'JPEG', 'PNG', 'JPG', 'WEBP’ 확장만 허용
- 이미지 리사이징 및 최적화
    - 가장 긴 변의 길이가 1024 이상인 경우 비율을 유지하며 1024로 축소

## 6.2. Competitor Analysis

> - 상품 카테고리를 기반으로 경쟁사 상품의 부정적인 리뷰를 크롤링
- 리뷰 요약과 상품 설명을 이용해 차별점 문장 생성
> 

**로컬-클라우드 분산 아키텍처 설계 및 구현**

- 크롤링 안정성과 서비스 응답 속도를 확보하기 위해 로컬 환경과 GCP 환경을 분리하여 구성
- MySQL DB를 매개로 로컬과 GCP 간 비동기 통신 및 역할 분담 수행
- GCP는 사용자 요청을 처리하고, 로컬은 크롤링을 담당
- 24시간 주기로 자동 크롤링 및 사용자 요청 대응을 병행하여 최신 데이터 유지

**로컬 환경**: 크롤링 및 리뷰 요약 처리

- VPN을 활용한 IP 우회를 통해 쿠팡의 크롤링 차단 회피
- 실행 시 멀티 스레드 구성
    1. 자동 크롤링 스레드 
        - 24시간마다 등록된 상품 카테고리에 대해 리뷰 재수집 및 요약 수행
        - 최신 리뷰 크롤링을 통한 데이터 신뢰도 확보
    2. 신호 기반 처리 스레드
        - GCP의 요청 신호를 5초 간격으로 감시
        - 요청 감지 시 해당 카테고리의 리뷰 수집 및 요약 수행
- 주요 기능 구성
    - 쿠팡 카테고리 검색 후 경쟁사 상위 3개 상품의 부정 리뷰 수집
        
        ```bash
        [1] 구매이유에어팟2세대 5년쓰니까 ...
        [2] 사용하던 버즈가 고장이 나서 구입했...
        [3] 4월26일에 구매했습니다!그리고 어제 고장...
        (생략)
        ```
        
    - GPT-4o를 이용한 불만 요약 키워드 추출
        
        ```bash
        [경쟁사 리뷰 요약]
        이 리뷰에서는 다음과 같은 불만, 단점, 개선점이 자주 언급되었습니다:
        
        1. 배터리 문제: 배터리 수명이 짧고, 갑작스럽게 방전되거나 충전이 되지 않는 문제가 있음.
        2. 연결 문제: 블루투스 연결이 불안정하여 소리가 끊기거나 자동으로 연결이 해제되는 경우가 많음.
        3. 음질 문제: 전반적으로 음질이 좋지 않으며, 특히 통화 시 음질이 떨어진다는 불만이 있음.
        (생략)
        ```
        
    - 요약 결과를 MySQL DB에 저장 또는 갱신

**GCP 환경**: 분석 요청 및 응답 처리

- 사용자 입력을 기반으로 실시간 경쟁사 분석 요청 처리
- 주요 처리 흐름 구성
    1. 사용자를 통해 상품 카테고리 및 상품 설명 확보
    2. DB에서 해당 카테고리의 리뷰 요약본 조회
    3. 요약본이 없을 경우 로컬 신호 테이블에 요청 신호 등록
    4. 5초 간격으로 DB를 확인하여 요약본을 확보
    5. 요약본 확보 후 자사 상품과 비교해 차별화 포인트 도출
    
    ```json
    "differences": [
        "지속적인 8시간 연속 재생으로 배터리 문제 해결",
        "안정적인 블루투스 연결로 끊김 문제 해결",
        "고감도 마이크로 통화 음질 개선",
        "IPX7 방수 등급으로 내구성 강화"
     ]
    ```

# **8. 📈 개발 이슈 및 해결 방안**

### ⚙️ **1. Settings**


**문제 정의 및 폴더 구조 정리**

- **문제**:  ‘소상공인을 위한 상세페이지 개발’이라는 추상적인 명제의 과제 목표로 혼란을 겪음
- **해결**: 팀원 전부 각자의 아이디어를 합쳐 회의를 통해 정리

**로깅에 대한 기준 부재**

- **문제**: 어느 부분을 어떻게 로깅을 할 지에 대한 기준이 없음
- **해결**: logging 라이브러리를 사용, 표준이 되는 약속을 정리

**VRAM 초과로 인해 서버 다운**

- **문제**: 허깅페이스 모델 다운로드 및 로딩 과정 중 VRAM초과로 인해  GCP 서버 다운
- **해결**: VRAM을 늘릴 수 없어 서버 재시작으로 해결

**DB 접근**

- **문제**: GCP 재부팅 시 IP 변경 → config.yaml 수정 및 커밋이 번거로움
- **해결**: DB IP를 .env에서 불러오도록 바꿔서 깃 커밋 없이 유연하게 관리

### 📂 **2. Input Handler**


**입력 데이터 유효성 문제**

- **문제**: 프론트에서 보내는 데이터가 예상한 스키마와 다를 경우, Pydantic이나 내부 검증에서 오류 발생
- **해결**: FormParser에서 강력한 정규화 & 예외 처리 필요

**이미지 경로 처리 오류**

- **문제**: 로컬 경로/URL 경로 구분이 명확하지 않거나(절대 경로, 상대경로), 파일이 존재하지 않을 경우 처리 중단
- **해결**: 파일 유효성 검사 로직을 ImagePreprocessor와 InputHandler 양쪽에 중복으로 두는 대신 **중앙 처리 유틸 함수**로 통일할 것

**config.yaml 저장 실패 또는 구조 오류**

- **문제**: 저장 중 인코딩 문제, 잘못된 데이터 타입 포함 시 YAML 파일 포맷이 깨지거나 저장 실패
- **해결** : `Dict[str, Any]` 구조 정합성 검증 후 `yaml.safe_dump(config, allow_unicode=True)`로 저장 처리

### 🌐 **3.** Competitor Analysis

**사이트의 봇 접근 차단**

- **문제**: 크롤링 대상 사이트가 봇 의심 접근을 차단하여 페이지에 접근할 수 없음
- **해결**: Selenium으로 브라우저를 직접 실행하고, fake_useragent를 사용해 일반 사용자처럼 접근

**사이트의 봇 의심 IP 차단**

- **문제**: 과도한 요청으로 IP가 임시 차단됨
- **해결**: VPN을 사용해 IP를 우회하고, 차단 시 다른 IP로 교체하여 크롤링 지속

**GCP 환경 내 VPN 사용의 제약**

- **문제**: GCP에서 VPN을 실행하면 IP가 변경되어 SSH 접속이 끊기거나 관리가 어려워짐
- **해결**: 크롤링은 local 환경에서 수행하고, 결과만 GCP로 연동하는 구조로 분리

**local 환경과 GCP 간 데이터 연동 문제**

- **문제**: local에서 크롤링한 데이터를 GCP에 어떻게 전달할지 불명확
- **해결**: GCP에 MySQL DB를 설치하고, local 환경에서 직접 DB에 접근하여 데이터 저장

**정적 리뷰 데이터로 인한 최신성 저하**

- **문제**: 리뷰 요약본을 DB에 저장해 사용하다 보니 최신 리뷰가 반영되지 않는 문제 발생
- **해결**: 저장된 상품 카테고리들을 주기적으로 다시 크롤링하여 최신 리뷰로 갱신

**사용자 요청 처리 로직 부재**

- **문제**: DB에 없는 카테고리를 사용자가 요청하면 처리 불가
- **해결**: 신호 테이블을 통해 GCP가 요청 신호를 보내면 local에서 감지 후 크롤링 수행

**DB에 데이터가 계속 축적됨**

- **문제**: `INSERT` 방식으로 저장하여 중복 데이터가 누적됨
- **해결**: `UPSERT` 쿼리(`ON DUPLICATE KEY UPDATE`)를 사용하여 기존 카테고리는 덮어쓰도록 변경

**local 실행 시 크롤링이 무조건 실행됨**

- **문제**: local 코드를 실행할 때마다 즉시 크롤링이 수행되어 주기 조건과 무관하게 동작함
- **해결**: DB에 저장된 마지막 크롤링 시각을 참조하여, 주기가 지난 경우에만 크롤링 수행

### 🖼️ 4. Image Generator


**모델 다운로드 실패 (디스크 부족)**

- **문제**: stabilityai/stable-diffusion-3.5-medium 다운로드 중 No space left on device에러
- **해결**: 불필요한 데이터 삭제로 30GB확보 후, 모델 변경(SDXL 기반 경량 모델)

**Stable Diffusion 2.1 사용시 의류 합성 실패**

- **문제**: 상품 이미지 기반 의류 착용 합성 불가능 → 인체 포즈 반영 안 됨
- **해결**: ControlNet(OpenPose) 적용 및 rembg로 배경 제거 후 재시도

**생성된 이미지 내 상품 디테일 왜곡**

- **문제**: SD 2.1에서 원본 상품 텍스처가 심하게 변형됨
- **해결**: IP-Adapter 기반 구조로 변경, 상품 특징 주입 방식 강화

**한글 프롬프트 비호환**

- **문제**: Stable Diffusion 모델이 한국어 프롬프트에서 품질 저하
- **해결**: 영어 기반 프롬프트로 변경, 번역 + 자연스러운 문장 구조 적용

**배경 복잡도 및 장면 중복**

- **문제**: 프롬프트 적용 시 배경이 복잡하고, 이미지에 2개의 장면이 합성되는 현상 발생
- **해결**: 프롬프트에 단일 scene 지시어 및 배경 단순화 지침 추가 → 개선 확인

**부정 프롬프트 효과 미비**

- **문제**: Negative Prompt 추가 후에도 생성 이미지의 차이가 미미
- **해결**: Negative Prompt 항목을 세분화(색상, 오브젝트, 로고 등)

**rembg 배경 제거 품질 불균일**

- **문제**: rembg로 배경 제거 시, 경계선이 거칠고 노이즈 발생
- **해결**: `alpha_matting` 활성화 + Threshold 조정 → 품질 개선

**프롬프트 생성기 품질 불안정**

- **문제**: 프롬프트 자동 생성 시 배경 설정이 지나치게 복잡하거나 어색함
- **해결**: 생성 프롬프트 길이를 15단어 이내로 단순화, 핵심 지시어 유지

**모델별 품질 차이 및 리소스 문제**

- **문제**: SD 3.5 품질 우수, 로드 시간·용량 부담 큼 / SDXL Turbo 속도 빠름, 일부 디테일 부족
- **해결**: 최종적으로 SDXL Turbo 채택 (속도·품질 균형)

**ControlNet 단독 성능 한계**

- **문제**: OpenPose/Canny 적용 후에도 자연스러운 합성이 어려움
- **해결**: ControlNet + IP-Adapter 병행 가능성 탐색, 스타일과 포즈 동시 반영

**SDXL Turbo 및 SDXL Base 이미지 품질 저조**

- **문제**: 배경 합성은 가능하지만 의류 착용 구현 불가능
- **해결**: SDXL Turbo를 상품 배경 생성 용도로만 활용, 사람+의류 합성은 인페인팅 기반으로 전환

**마스크 기반 인페인팅 시도**

- **문제**: Stable Diffusion 2 Inpainting 사용 시 테두리 불명확, 자연스러움 부족
- **해결**: 마스크 정교화 전략 검토 → 프로토타입 수준에서 제한적 활용

**이미지 생성 전략 재정립**

- **문제**: 완전한 VTON 구현은 시간·리소스 한계로 어려움
- **해결**: **단기 전략**: IP-Adapter + SDXL 조합으로 프로토타입 완성
          **장기 전략**: IDM-VTON 또는 TryOnDiffusion 활용, ControlNet과 FaceID 연동 고려

**IP-Adapter 기반 사람 이미지 품질 불안정**

- **문제**: 손가락 등 일부 디테일 비현실적
- **해결**: 추론 횟수 15→50회로 증가 및 이미지 크기 768×768로 조정

**IP-Adapter + SDXL 사용 시 상품 강조 부족**

- **문제**: 배경에 비해 상품이 잘 부각되지 않음
- **해결**: 프롬프트에 `photorealistic`, `product-focused`, `studio lighting`등 구체 키워드 삽입

**VTON 사용 시 마스킹 적용 문제**

- **문제**: 정밀 마스킹 자동화 어려움 → 일부 이미지에서 합성 품질 저하
- **해결**: 초기에는 수동 마스크 적용, 추후 자동화 기능 탐색 예정

**이미지 객체 반환 오류**

- **문제**: FastAPI 응답 시 `PIL.Image` 객체는 JSON 직렬화 불가 → API 응답 실패
- **해결**: 이미지 경로(path)를 반환하는 방식으로 수정

**시드 값 고정으로 인한 이미지 다양성 부족**

- **문제**: 동일한 시드 값으로 매번 유사 이미지 생성 → 결과물이 반복적
- **해결**: 날짜 기반 랜덤 시드(년월일시분초) 적용

**IP-Adapter Virtual Try-On 정확도 부족**

- **문제**: 초기 시도 시 의류 위치가 어색하고 사람 이미지 품질 낮음
- **해결**: HuggingFace 블로그 기반 파이프라인 재구성 및 파라미터 조정으로 개선

**원본 의류 영향으로 합성 불안정**

- **문제**: 기존 옷이 남아있으면 새로운 옷 합성 자연스럽지 않음
- **해결**: 옅은 의상 착용 이미지 사용 + 마스킹으로 타겟 구역 분리

**마스킹 작업 자동화 미비**

- **문제**: 상의나 특정 부위만 정확히 마스킹하는 자동화 기능 부재 → 반복 작업 비효율
- **해결**: 사용자가 인물 이미지와 마스킹 이미지를 선택하는 방식으로 대응

**이미지 생성 속도 문제**

- **문제**: IP-Adapter + ControlNet + Inpainting + LoRA 조합 시 평균 생성 시간 2분 30초 소요
- **해결**: 모델 최초 1회 로드, 추론 횟수 감소로 30초 이내로 단축

**인물 이미지 자세에 따른 품질 편차**

- **문제**: 모델 자세가 비대칭이면 합성 왜곡 발생
- **해결**: 정자세 모델 이미지를 선별해 적용

**VTON 자체의 이미지 품질 낮음**

- **문제**: Virtual Try-On 시 인체 비율 및 상품 디테일 손상
- **해결**: VTON API 활용으로 정확도 높은 결과 확보

- **해결**: OpenAI 생성 결과 50개를 학습 데이터로 활용하여 LoRA 방식으로 추가 학습을 진행
- **효과**: 문장 유창성이 향상되었으며, 전문가 인용 파트가 생기는 등 GEO 최적화 지시를 반영

- **해결**: 프롬프트에 Tailwind CSS를 사용하도록 명시하여 OpenAI가 HTML + CSS 전체를 완성도 높게 생성하도록 변경
- **효과**: 실제 상품 상세페이지와 유사한 결과물로 시각적 완성도 확보

**CSS 디자인의 일관성 문제**

- **문제**: OpenAI가 생성하는 CSS가 요청할 때마다 달라져 디자인 일관성 저하
- **해결**: 특정 HTML 태그에 적용할 CSS 클래스를 프롬프트에 지정 (예: 고객 정보는<p class="font-semibold text-blue-700">로 표시할 것)

**이미지 태그의 AI 검색 노출 최적화**

- **문제**: 기존 HTML 내 이미지 태그에는 alt 속성이 없어 AI 검색 노출 최적화가 부족
- **해결**: 상품 특징을 바탕으로 alt 속성을 생성하도록 프롬프트를 수정

### 🚧 7. Backend


**Form + UploadFile 병합 처리 이슈**

- **문제**: Form과 File을 함께 받을 때 클라이언트에서 multipart/form-data를 올바르게 보내지 않으면 FastAPI가 파싱 실패함.
- **해결**: 프론트엔드에서 multipart/form-data 제대로 구성 확인

**UploadFile 처리 및 유효성 검사**

- **문제**:
    - UploadFile은 임시파일로 저장되므로, `.file.read()` 혹은 `.save()` 시점의 메모리/디스크 이슈 발생 가능
    - 빈 파일이거나 이미지가 아닌 파일 업로드 시 처리 누락 가능성
- **해결**: filename, content_type 체크 및 이미지 유효성 검사 수행

**입력 파라미터 유효성 검증 미흡 시 오류 전파**

- **문제**: int, str 타입의 파라미터에 부적절한 값이 들어오면 자동으로 `422 Unprocessable Entity` 발생
- **해결**: 프론트엔드에서 정제된 입력만 보내도록 강제하고, Pydantic 모델 추가 사용

**InputHanlder, ImageComposer 의존성 주입 시 에러**

- **문제:** 객체 내부에서 exception 발생해도 FastAPI는 Depends 주입 시 무조건 해당 함수를 먼저 실행
- **해결**: 생성자 내부에서 에러 발생 시 try/except 로 예외 처리 → API에서 명확히 실패 사유 반환

**비동기 처리시 `asyncio.gather()` 에러 전파 이슈**

- **문제**: `asyncio.gather()` 내부에서 발생한 에러는 gather 자체가 ExceptionGroup으로 감싸서 반환 → 디버깅 어려움
- **해결**:
    - `return_exceptions=True` 사용 시 결과 필터링 필요
    - 내부 함수에 `try/except` 명확히 분리

**후보 이미지 처리 구조 불일치 문제**

- **문제**:
    - `generate_image()` 결과가 딕셔너리일 수도 있고 리스트일 수도 있음
    - 이로 인해 `product['candidate_images']`가 `[[]]`, `[[path1, path2]]` 등 혼동될 수 있음
- **해결**:
    - 반환 형식 고정 (항상 dict로 감싸서 `{ "image_paths": [...] }` 형태로 통일)

### 🖱️ 8. Frontend


**입력 유효성 검사 누락**

- 문제: 필수 입력값이 비어 있는데도 제출 버튼이 눌리거나, 잘못된 값이 그대로 전달되는 경우
- **해결**:
    - `if not value:` 또는 `st.warning()` 활용한 클라이언트 검증 추가
    - 숫자형 필드는 `st.number_input()`처럼 타입 강제 사용

**세션 상태 꼬임(st.session_state)**

- 문제: 한 페이지에서 설정된 상태가 다른 페이지에서 예상과 다르게 동작하거나 초기화됨
- **해결**:
    - 세션 키 존재 여부 검사 (`if "processed_data" not in st.session_state`)
    - 모든 페이지에서 세션 키 기본값 초기화

**파일 업로드 후 경로 처리 문제**

- 문제: 업로드된 이미지의 경로 또는 데이터가 잘못 처리됨
- **해결**: 파일 임시 저장 시 BytesIO 또는 temp 디렉토리 사용

**REST API 요청 실패 및 응답 처리 문제**

- 문제: 백엔드와의 통신 중 오류가 발생했을 때 사용자에게 피드백이 부족함
- **해결**: HTTP 응답코드 기반 분기 처리 (200 / 400 / 500 등)

**페이지 이동 흐름 혼동(result.py 진입 조건 오류)**

- 문제: 필수 입력 또는 이전 단계 처리 없이 결과 페이지 진입 시 에러 발생
- **해결**: 모든 페이지에 진입 조건 명시 (`if "xxx" not in st.session_state: st.warning("먼저 입력을 완료해주세요")`)

**이미지 미리보기 or 다운로드 문제**

- 문제: 이미지 URL 또는 바이트 데이터 처리 오류로 미리보기가 안 보이거나 다운로드가 깨짐
- **해결**:
    - 이미지 URL 또는 바이트 여부에 따라 분기 처리
    - 다운로드 시 BytesIO + MIME 타입 설정 필수

# **9. 📌 결론**

## **프로젝트 목표 달성 여부**

본 GEOPage 프로젝트는 소상공인을 위한 **AI 기반 상세페이지 자동 생성 도구**를 성공적으로 구현하였습니다. 사용자가 입력한 상품 정보를 바탕으로 GPT를 활용해 **GEO·SEO 최적화 텍스트**를 생성하고, **RealVisXL, Gemini API 기반의 이미지 생성**까지 통합하여 상세페이지 제작의 전 과정을 자동화했습니다. FastAPI와 Streamlit을 활용한 구조를 통해, 누구나 손쉽게 사용할 수 있는 서비스 형태로 완성되었습니다.

- **GEO 최적화 상세페이지 생성**
    
    LLM 모델을 기반으로 경쟁사 대비 차별점이 드러나는 GEO 상세페이지 자동 생성에 성공했습니다. 실제 리뷰 기반 분석과 결합해 고유한 마케팅 포인트를 도출할 수 있게 구성했습니다.
    
- **AI 이미지 생성 파이프라인 구현**
    
    ControlNet 기반의 의류 착용샷 생성 워크플로우를 성공적으로 구현했으며, 해당 기능을 사용자 인터페이스에 통합했습니다.
    
- **비개발자도 사용할 수 있는 웹 도구 개발**
    
    Streamlit 인터페이스를 통해 누구나 접근 가능한 웹 서비스를 제공했으며, API 기반 FastAPI 백엔드를 구성하여 확장성과 유지보수성을 고려한 구조를 갖췄습니다.
    
- **의류 도메인 실험 적용**
    
    실제 의류 상품 예시를 기반으로 차별점 도출 → 이미지 생성 → 상세페이지 완성까지 전체 파이프라인을 시험하며, 도메인 특화 가능성을 검증했습니다.
    

## **한계점 및 개선 방향**

- **VTON 자동화 실패**
    - **한계점**: Hugging Face 기반 VTON 파이프라인에서 마스킹 자동화가 불완전했고, 생성 이미지의 품질 역시 낮아 실용적 사용에 실패했습니다. Fine-tuning 없이 IP-Adapter + LoRA 조합만으로는 의류 이미지의 복잡한 스타일을 반영하는 데 한계가 있었습니다.
    - **개선 방향**: 마스킹 정확도를 높이기 위해 Segment Anything 등의 도입을 고려할 수 있으며, 의류 데이터셋 확보 후 ControlNet 또는 DreamBooth 기반의 Fine-tuning을 통해 도메인 특화 성능을 향상시킬 수 있습니다.
- **리소스 제약으로 인한 확장성 부족**
    - **한계점**: 15GB 이상의 VRAM과 10GB 이상 디스크 용량을 요구하는 모델을 로컬 환경에서 실행하는 데 어려움이 있었습니다. 이에 따라 실시간 서비스 수준으로는 구현이 불가능했습니다.
    - **개선 방향**: ONNX, TensorRT 기반의 양자화(Quantization) 또는 모델 경량화 전략을 실험하여 로컬에서도 최소한의 성능을 확보하거나, 클라우드 GPU 인스턴스 임대를 통해 확장성을 확보할 수 있습니다.
- **정량적 품질 측정 부재**
    - **한계점**: 이미지와 상세페이지 생성 결과에 대해 CLIP Score, FID, PSNR 등 객관적인 품질 지표 없이 정성적 평가에만 의존했습니다. 사용자 선호도나 비교 테스트도 진행되지 않았습니다.
    - **개선 방향**: 생성 결과에 대한 CLIP Score, FID 등 자동 품질 평가 도구를 도입하고, 사용자 피드백 기반의 A/B 테스트로 응답 품질을 수치화할 수 있는 환경을 구성할 필요가 있습니다.
- **실패 사례에 대한 기록 부족**
    - **한계점**: 허깅페이스 이미지 생성 중 손가락 왜곡, 디테일 깨짐, 마스킹 실패 등이 빈번했으나, 이와 같은 실패 케이스에 대한 구체적인 발생률이나 유형 통계는 남기지 못했습니다.
    - **개선 방향**: 실패 케이스에 대한 로그 및 예시 이미지를 자동 수집/분류하고, 주요 오류 유형에 대한 통계화를 통해 개선 방향을 분석할 수 있는 기반을 마련할 수 있습니다.
- **서버 자원 부족 시 서비스 불안정**
    - **한계점**: 사용자 다수의 동시 요청 시 서버 리소스가 부족해 서비스가 중단되는 현상이 발생했습니다.
    - **개선 방향**: 요청 큐 처리 방식 도입(Celery, Redis Queue 등)을 통해 병렬 요청에 대응하고, 자주 요청되는 결과를 캐시 처리하며, GPU 인프라의 autoscaling 환경(GCP Cloud Run, Lambda GPU 등)도 고려할 수 있습니다.
